Cloudflare 把后量子目标提前到 2029：现在更危险的，已经不是“以后再解密”，而是“认证先被打穿”

    Cloudflare把完整后量子安全迁移目标提到2029年，Google也同步加速时间表；真正紧迫的不只是旧流量未来被解密，而是认证、证书和签名链路先被量子能力伪造。对团队而言，长期密钥、API凭证和旧算法清理应优先推进。

    这两年一提 “后量子密码迁移”，很多人的直觉还是：这是迟早要做，但还没到今天就得动手的事。

    Cloudflare 在 4 月 7 日把这个气氛往前拽了一大截。它最新宣布的目标是：到 2029 年实现完整的后量子安全，而且重点包括认证（authentication）这一层。

    这条消息值得写，不只是因为 Cloudflare 又更新了一张时间表，而是因为它释放了一个更强的行业信号：

    后量子风险的焦点，正在从 “未来有人能不能解密今天截获的数据”，转向 “认证体系会不会先成为入口”。

    # 这次更新到底变了什么？

    按 Cloudflare 官方博客的说法，它此前已经在 2022 年开始默认推进后量子加密，并称如今流向 Cloudflare 的人类流量里，已经有 65% 以上 使用了后量子加密。

    但这次真正被提前的，是更难、也更关键的那一部分：后量子认证。

    Cloudflare 这次直接把目标定在 2029 年前完成全产品线的后量子安全迁移，并明确强调：

    • 只做后量子加密还不够；
    • 证书、签名、身份验证、访问凭证这些 “认证相关能力” 也必须尽快迁移；
    • 迁移不只是 “支持新算法”，还包括后续关闭脆弱旧算法、避免 downgrade attack，以及轮换曾暴露在旧体系下的密钥和秘密。

    这和过去很多团队对 PQC 的理解不太一样。以前大家更多把它看成一个 “先补加密，认证以后再说” 的长期任务；现在 Cloudflare 的口径已经很明显：如果 Q-Day 比预期更近，认证风险会先变成真正致命的问题。

    # 为什么 Cloudflare 会突然把时间线拉得这么靠前？

    核心背景有两个。

    第一，Google 也把自己的后量子迁移时间线压到了 2029。
    Google 官方博客这次明确说，随着量子硬件、纠错和算法进展加快，威胁模型已经需要调整，尤其要优先考虑认证服务的迁移。

    第二，Google Research 在 3 月 31 日公开的新研究里，更新了对 ECDLP-256 的量子资源估算。按官方表述，在其假设下，未来的密码学相关量子计算机破解椭圆曲线密码所需资源，比之前的理解又往前推了一大步，甚至给出了少于 50 万物理量子比特、数分钟级执行的估算路径。

    这不等于 “明天就会被打穿”，但它足以改变工程判断：
    如果时间窗口不再是十多年，而是几年的量级，那迁移顺序就必须重排。

    # 真正值得在意的：为什么 “认证” 比 “解密旧流量” 更急？

    这是我觉得这条新闻最值得记住的地方。

    过去行业最常提的是 harvest now, decrypt later—— 攻击者今天先把加密流量存下来，等未来量子能力成熟后再解密。这当然很严重，尤其对长期敏感数据来说更是如此。

    但 Cloudflare 这次强调的是另一层：一旦量子攻击足以伪造签名、冒充证书或破解长期认证密钥，问题就不再只是 “数据会泄露”，而是 “系统入口会被直接接管”。

    比如：

    • 长生命周期证书被伪造；
    • API 认证密钥被量子能力击穿；
    • 远程登录密钥失去可信性；
    • 软件更新签名体系被绕过。

    这类后果和 “旧流量未来被解密” 相比，破坏性往往更直接。前者是前门被打开，后者更多是保险箱被事后撬开。

    也正因为这样，Cloudflare 和 Google 现在都在把话题往 “认证优先” 上推。

    # 这对技术团队意味着什么？

    如果你在做平台、安全、基础设施或企业 IT，这条新闻其实已经不是 “科普新闻”，而更像一个路线图提醒。

    # 1. 后量子迁移不再只是密码学团队的事

    它会波及证书体系、SSO、API 网关、代码签名、CI/CD、供应链更新机制，甚至外部供应商接入方式。换句话说，这是组织级工程问题，不是单点组件升级。

    # 2. 最该先排查的是 “长寿命、高权限” 密钥

    不是所有系统都要同一天完成迁移，但优先级已经很清楚：

    • 根证书和中间证书
    • 代码签名与更新签名
    • 高权限 API 凭证
    • 长期有效的远程访问密钥
    • 关键第三方依赖里的认证链路

    这些东西一旦出事，影响不是单点泄露，而是整条信任链断掉。

    # 3. “支持 PQ” 不是终点

    Cloudflare 这次有一句很关键：仅仅添加后量子支持并不够。
    如果旧算法仍然长期保留、旧密钥继续复用、旧秘密没有轮换，那系统仍然可能被降级攻击或历史暴露拖后腿。

    很多迁移项目最容易卡死的地方，也恰恰在这里：
    不是新东西装不上，而是旧东西关不掉。

    # 小结

    所以，这条消息最值得记住的一句话，我会写成：

    后量子安全的真正倒计时，开始从 “什么时候有人能解密旧数据”，转向 “什么时候有人能伪造你的信任入口”。

    Cloudflare 把目标提前到 2029，本质上是在提醒整个行业：
    PQC 已经不是一件可以无限往后排的 “未来事项”，而是正在进入架构规划、认证治理和供应链安全的当前议题。

    接下来几年，真正成熟的团队，拼的可能不只是 “有没有开始支持后量子算法”，而是：
    能不能把认证、签名、密钥轮换和第三方依赖一起迁过去。
