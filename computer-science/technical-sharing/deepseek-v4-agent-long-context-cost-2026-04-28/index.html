DeepSeek-V4 发布后，我更在意的不是“1M 上下文”，而是开源 Agent 模型终于开始围着长任务成本来设计

    DeepSeek V4 不只在 100 万 token，更把长上下文做得更省：1M 场景推理算力仅 V3.2 的 27%，KV 缓存仅 10%。配合混合注意力和 Agent 评测，说明开源模型已把长任务延迟、显存和稳定性当产品级优化重点。

    DeepSeek 放出 V4 之后，最容易抢眼球的当然是 100 万 token 上下文。

    但如果只把它理解成 “上下文又变长了”，我觉得会低估这次发布真正值得记的一点：

    开源模型终于开始认真回答 Agent 时代的硬问题：长任务不是能不能塞进上下文，而是塞进去以后还跑不跑得动。

    # 1. 这次最关键的不是 “更长”，而是 “更便宜地长”

    按 DeepSeek 官方模型卡，DeepSeek-V4-Pro 在 1M token 场景下，相比 V3.2 只需要：

    • 27% 的 single-token inference FLOPs；
    • 10% 的 KV cache。

    这两个数字比 “支持 1M context” 本身更重要。

    因为真正跑过长链路 Agent 的人都知道，问题通常不是第一轮放不下，而是：

    • 工具调用结果会越积越多；
    • 后面每一个 token 都要为前面的历史买单；
    • 上下文一深，延迟和显存就会一起涨。

    所以 V4 给出的核心信号是：长上下文开始从 marketing 数字，变成推理成本工程。

    # 2. 它不是只在卷窗口，也在卷 “能不能把 Agent 跑完”

    这次 DeepSeek-V4 系列一共给了两档：

    • V4-Pro：1.6T 总参数，49B 激活参数；
    • V4-Flash：284B 总参数，13B 激活参数。

    两者都支持 1M 上下文，但官方更想强调的显然不只是 “能装很多字”，而是这种长度对真实任务是否有意义。

    官方列出的 V4-Pro Max Agent 相关成绩里，有几项很有代表性：

    • Terminal Bench 2.0：67.9
    • SWE Verified：80.6
    • MCPAtlas Public：73.6
    • Toolathlon：51.8

    这些分数当然还不是把所有闭源前沿模型都压过去，但已经很能说明问题：

    开源模型现在争的，不再只是聊天强不强、题做得漂不漂亮，而是能不能稳定做长任务、能不能把工具链真正接起来。

    # 3. 我更在意的是：它开始围着 Agent 工作负载做底层适配

    DeepSeek 官方把这次架构升级的重点放在 Hybrid Attention，也就是 CSA + HCA 的组合上。对大多数开发者来说，机制细节不一定要逐层吃透，但有一件事很明确：

    它瞄准的是 长上下文的可用成本，不是单纯把窗口数字写大。

    这件事的意义很现实。

    对普通聊天产品来说，上下文长一点短一点，很多时候只是体验差异；但对 Agent 来说，长上下文往往意味着：

    • 多轮计划；
    • 多次工具往返；
    • 更长的错误恢复链路；
    • 更高的持续运行成本。

    所以当一个开源模型把 “百万上下文下还能不能省着跑” 当成一等公民，它其实是在朝另一类工作负载靠拢：

    不是一次性问答，而是长时间执行。

    # 我的判断

    DeepSeek-V4 这次最值得记住的，不是 “开源也有 1M context 了”，而是：

    开源 Agent 模型终于开始把延迟、显存和长链路稳定性，当成产品级问题来解决。

    如果这个方向继续往前走，接下来真正会卷的，可能不只是 benchmark 排名，而是三件更硬的事：

    • 长链路任务跑得稳不稳；
    • 工具调用和上下文续接顺不顺；
    • 同样的 Agent 工作量，到底烧多少算力。

    这比单纯把上下文窗口继续往上堆，更值得看。
